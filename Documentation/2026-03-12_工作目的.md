# 工作目的 (2026-03-12)

> 说明本轮「为什么做这些事」，便于后续对话或新成员理解。改动列表见 `2026-03-12_项目改动清单.md`，代码细节见 `2026-03-12_代码改动说明.md`。

## 2026-03-12 工作目的

- **目的一：统一 Manager 单例与跨场景逻辑**
  - 通过新增 `Assets/Scripts/Common/Singleton.cs`，将 AccountManager、SaveManager、UIPanelManager 等原本各自维护的单例与 `DontDestroyOnLoad` 逻辑抽象到一个可复用的基类中，减少重复代码和潜在 bug。  
  - 后续若再新增其它需要跨场景存在的管理类，只需继承 `Singleton<T>` 即可，保证风格与行为一致。  

- **目的二：把「标题 → 登录/注册 → 存档选择 → GameScene」流程打磨为可直接使用的基础框架**
  - 在 3 月 10 日基础上进一步完善 UI 与服务层交互逻辑：  
    - LoginPanel 在登录失败时不再误跳转到 SaveSelectPanel，同时加入注册功能，使账号流程在 UI 层完整闭环；  
    - SaveSelectPanel 正确区分「空槽位」与「已有存档」：空槽位通过 `CreateDefaultSaveInSlot` 创建默认存档，已有存档则 SelectSlot 并加载 `GameScene`，避免之前 SelectSlot/HasSaveInSlot 顺序导致的逻辑混乱；  
    - 所有面板名称通过常量/序列化字段配置，避免硬编码与拼写错误，让 UI 流程修改更安全。  

- **目的三：清理并统一相关脚本中的中文显示与日志**
  - 按 `.cursor/rules/code-chinese-format.mdc` 规范，逐一检查本轮涉及的服务类与 UI 类，修复 `LoginPanel`、`SaveSelectPanel`、`AccountEntry` 等脚本中的中文乱码，统一为 UTF-8 简体中文。  
  - 通过统一中文日志内容（如「AccountManager为空!」「登录失败!」「该位置没有存档! 将创建默认存档。」等），为日后调试与排查问题提供更友好的信息。  

- **目的四：为后续 UI 与玩法扩展打好地基**
  - 经由本轮改造后，UI 层已经具备：  
    - 通用面板基类 + 面板管理器；  
    - 配置化的面板名称管理；  
    - 稳定可用的账号登录/注册 + 存档选择基础流程。  
  - 后续无论是增加设置面板、错误提示弹窗，还是扩展更多游戏内 UI，都可以复用本轮搭建的框架，减少重复劳动。
