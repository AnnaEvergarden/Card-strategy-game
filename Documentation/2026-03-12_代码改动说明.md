# 代码改动说明 (2026-03-12)

> 逐一说明每处代码改动的**作用**、**改动前**、**改动后**、**目的**。改动清单与工作目的见 `2026-03-12_项目改动清单.md`、`2026-03-12_工作目的.md`。

## 1. Singleton\<T\>（新建）

- **作用**：  
  通用的 MonoBehaviour 单例基类，统一提供线程安全的 `Instance` 访问、重复实例自毁、可选跨场景持久化等能力，为所有需要跨场景存在的管理类（Manager）复用。

- **改动前**：  
  - SaveManager、AccountManager、UIPanelManager 等类各自实现单例逻辑（静态 Instance + Awake + DontDestroyOnLoad），写法不统一，维护成本高。  
  - 修改任一 Manager 的生命周期行为（如是否跨场景）都需要单独改写该脚本。  

- **改动后**（`Assets/Scripts/Common/Singleton.cs`）：  
  - `public abstract class Singleton<T> : MonoBehaviour where T : MonoBehaviour`：要求 T 为 MonoBehaviour。  
  - 静态字段 `_instance` + `_lock`，`Instance` 属性内部双重检查锁，确保线程安全地初始化引用（虽然在 Unity 中多线程访问较少，但规范性更好）。  
  - `protected virtual bool PersistAcrossScenes => true;`：子类可通过重写该属性控制是否 `DontDestroyOnLoad`。  
  - `protected virtual void Awake()`：  
    - 若 `_instance` 已存在且不是当前对象，`Destroy(gameObject)`，防止重复单例；  
    - 否则将当前对象缓存到 `_instance`，并在 `PersistAcrossScenes == true` 时调用 `DontDestroyOnLoad(gameObject)`。  
  - `protected virtual void OnDestroy()`：若当前对象即 `_instance`，在销毁时将 `_instance` 置空。  

- **目的**：  
  收敛所有 MonoBehaviour 单例的实现方式，提高一致性与可维护性，为后续扩展更多跨场景管理类提供统一基础。

## 2. UIPanelManager（修改）

- **作用**：  
  集中管理所有 UI 面板（BaseUIPanel 子类），负责：  
  - 注册各面板；  
  - 按名称显示指定面板并隐藏其他面板；  
  - 按需提供面板引用（GetPanel）。  

- **改动前**（旧版大致行为）：  
  - 自行维护静态 `Instance` 与 `Awake`，与其他 Manager 写法不统一。  
  - `ShowPanel(string _panelName)` 在找不到目标面板时，会错误地再次调用 `UIPanelManager.Instance.ShowPanel(_panelName)`，导致潜在的无限递归风险。  

- **改动后**（`Assets/Scripts/Services/UIPanelManager.cs`）：  
  - `public class UIPanelManager : Singleton<UIPanelManager>`：改为继承通用单例基类。  
  - 删除类内自定义的 `Instance` 与 `Awake`，统一由 `Singleton<T>` 管理生命周期。  
  - 内部使用 `Dictionary<string, BaseUIPanel> panels` 存储面板注册信息不变。  
  - `RegisterPanel(string _panelName, BaseUIPanel _panel)`：  
    - 若 `_panelName` 为空或 `_panel` 为 null，则输出警告并返回；  
    - 合法情况下将其存入字典，并调用 `_panel.HidePanel()`，保证初始状态为隐藏。  
  - `ShowPanel(string _panelName)`：  
    - 先 `foreach` 所有已注册面板并调用 `HidePanel()`，确保切换前清空显示状态；  
    - 然后使用 `panels.TryGetValue(_panelName, out var _panel)` 获取目标面板：  
      - 找到则调用 `_panel.ShowPanel()`；  
      - 找不到则输出 `"{_panelName} 不存在！"` 的警告，不再递归调用自身。  
  - `GetPanel(string _panelName)` 通过 `TryGetValue` 安全返回对应面板或 null。  

- **目的**：  
  - 去除重复单例实现，统一使用 `Singleton<T>`；  
  - 修复 ShowPanel 的潜在递归 bug；  
  - 保证任何时候只有一个面板处于显示状态，避免 UI 状态混乱。

## 3. BaseUIPanel + UIPanelNames（修改）

- **作用**：  
  - `BaseUIPanel`：所有 UI 面板的抽象基类，负责注册自身到 UIPanelManager，并提供统一的 Show/Hide 行为；  
  - `UIPanelNames`：集中维护常用面板名称常量，避免散落的字符串硬编码与拼写错误。  

- **改动前**：  
  - BaseUIPanel 使用 `GetPanelName()` 返回 `gameObject.name` 作为面板名，依赖物体命名，耦合较高。  
  - 注册与默认隐藏逻辑不够统一，一些子类需要自己管理 SetActive。  
  - 未有集中管理面板名称的静态类。  

- **改动后**（`Assets/Scripts/UI/BaseUIPanel.cs`）：  
  - 新增静态类 `UIPanelNames`：  
    - `public const string Title = "TitlePanel";`  
    - `public const string Login = "LoginPanel";`  
    - `public const string SaveSelect = "SaveSelectPanel";`  
  - `public abstract class BaseUIPanel : MonoBehaviour`：  
    - 新增 `[SerializeField] public string PanelName;`，通过 Inspector 显式配置本面板 ID。  
    - `public void Awake()`：在 Awake 中调用 `RegisterPanel()` 和 `HidePanel()`：  
      - 确保面板在场景加载时自动注册到 UIPanelManager；  
      - 并默认处于隐藏状态，避免「一进场景就全部显示」。  
    - `public virtual void RegisterPanel()`：  
      - 通过 `UIPanelManager.Instance` 获取管理器；  
      - 当 UIManager 或 PanelName 为空时输出 `"UIManager为空或面板没有命名!"`；  
      - 否则调用 `uiManager.RegisterPanel(PanelName, this)` 完成注册。  
    - `public virtual void ShowPanel()` / `HidePanel()`：分别调用 `gameObject.SetActive(true/false)`。  
    - 预留 `protected virtual void OnShow()` / `OnHide()` 供子类在展示/隐藏时追加逻辑（当前版本未使用）。  

- **目的**：  
  - 使 UI 面板的注册与隐藏行为完全统一，并尽量减少对子类的要求；  
  - 把面板 ID 显式化（常量 + 序列化字段），方便未来改名与调试；  
  - 为后续新增面板（设置、弹窗等）提供标准模板。

## 4. TitlePanel（修改）

- **作用**：  
  标题界面，提供「登录」「登出」「设置」「退出游戏」四个入口按钮。  

- **改动前**：  
  - 点击「登录」时，直接在代码中硬编码要切换的面板名称，如 `"Login"`，与实际面板名或常量不易保持同步。  

- **改动后**（`Assets/Scripts/UI/TitlePanel.cs`）：  
  - 新增：  
    - `[Header("引用面板")]`  
    - `[SerializeField] public string _LoginPanelName;`：在 Inspector 中配置目标登录面板 ID（建议使用 `UIPanelNames.Login`）。  
  - `OnClickLogin()` 改为：`UIPanelManager.Instance.ShowPanel(_LoginPanelName);`。  
  - 登出、设置占位、退出游戏行为保持原有逻辑。  

- **目的**：  
  - 消除硬编码字符串依赖，避免未来改名时忘记同步导致的「按钮点击无效」问题；  
  - 把 UI 流程的可配置点更多地交给 Inspector。

## 5. LoginPanel（修改）

- **作用**：  
  账号登录/注册界面，负责从 TMP_InputField 读取用户名与密码，调用 AccountManager 完成登录/注册，并根据结果切换 UI 面板。  

- **改动前**：  
  - 空引用检查时日志文本错误（提示 SaveManager 而非 AccountManager），且存在中文乱码。  
  - 调用 `accountManager.Login()` 后即使返回 false 仍会切换到存档选择面板，逻辑有误。  
  - 目标面板名硬编码，且存在拼写错误（如 `"SaveSelcet"`）。  
  - 仅提供登录功能，无注册按钮逻辑。  

- **改动后**（`Assets/Scripts/UI/LoginPanel.cs`）：  
  - 新增面板名配置字段：  
    - `[SerializeField] public string _SaveSelectPanelName;`：登录成功后要展示的存档选择面板名；  
    - `[SerializeField] public string _TitlePanelName;`：取消或返回时要展示的标题面板名。  
  - `Start()`：  
    - 若 `accountManager == null`，输出 `Debug.LogWarning("AccountManager为空!");` 并 `return`。  
  - `OnClickLogin()`：  
    - 从 `userNameInput` / `passwordInput` 取文本；  
    - 调用 `accountManager.Login(userName, password)`：  
      - 若返回 false，输出 `Debug.Log("登录失败!");` 并 `return`；  
      - 若成功，则调用 `UIPanelManager.Instance.ShowPanel(_SaveSelectPanelName);` 进入存档选择。  
  - 新增 `OnClickRegister()`：  
    - 同样先读取用户名与密码；  
    - 若 `accountManager.Login(userName, password)` 已成功，说明账号已存在，则输出 `Debug.Log($"{userName} 该用户已存在并成功登录!");`；  
    - 否则调用 `accountManager.Register(userName, password)` 创建新账号，并输出 `Debug.Log($"{userName} 注册成功！");`。  
  - `OnClickCancel()`：调用 `UIPanelManager.Instance.ShowPanel(_TitlePanelName);` 返回标题界面。  
  - 所有中文日志文字全部改为 UTF-8 简体（无乱码）。  

- **目的**：  
  - 修复登录失败仍跳转的问题，确保 UI 行为与账号状态一致；  
  - 补齐注册功能，让账号流程在 UI 层自洽；  
  - 清除中文乱码，便于后续排错与查看日志。

## 6. SaveSelectPanel（修改）

- **作用**：  
  为当前账号展示固定数量的存档槽位按钮，玩家可以选择已有存档进入游戏，或在空槽位上创建默认存档。  

- **改动前**：  
  - 未继承 BaseUIPanel，UI 流程风格不统一。  
  - 在 `for` 循环中直接使用 `i` 绑定 `onClick`，导致所有按钮在点击时共享最后一个索引（闭包 bug）。  
  - 点击槽位时先调用 `SelectSlot`（内部会 `Load` 并可能创建默认存档），再检查 `HasSaveInSlot`，从逻辑上几乎总是「已有存档」，空槽位分支形同虚设。  
  - `SceneManager.LoadScene("Game")` 与实际场景名 `GameScene` 不一致。  
  - 部分中文日志乱码或不规范。  

- **改动后**（`Assets/Scripts/UI/SaveSelectPanel.cs`）：  
  - 类头改为 `public class SaveSelectPanel : BaseUIPanel`。  
  - `Start()`：  
    - 若 `saveManager == null`，尝试 `SaveManager.Instance` 赋值；  
    - 若 `container`、`slotPrefab` 或 `saveManager` 任一为空，输出 `"[SaveSelectPanel] container / slotPrefab / saveManager 未赋值或未找到！"` 并 return；  
    - 清除 `container` 下现有子节点；  
    - 使用 `for (int i = 0; i < SaveManager.SaveSlotCount; i++)` 动态生成槽位：  
      - 为每次循环声明本地变量 `int slotIndex = i;`，并在 `btn.onClick.AddListener(() => OnClickSlot(slotIndex));` 中使用，修复闭包 bug；  
      - 通过 `saveManager.HasSaveInSlot(slotIndex)` 决定显示「有/无」存档状态。  
  - `OnClickSlot(int _slotIndex)`：  
    - 若 `saveManager == null`，日志 `"SaveManager为空！"` 后 return；  
    - 若 `!saveManager.HasSaveInSlot(_slotIndex)`：  
      - 调用 `saveManager.CreateDefaultSaveInSlot(_slotIndex)` 创建默认存档；  
      - 输出 `Debug.LogWarning($"该位置没有存档! 将创建默认存档。");` 并 return（暂时停留在当前界面）。  
    - 若存在存档：  
      - 调用 `saveManager.SelectSlot(_slotIndex)`；  
      - 输出 `"即将加载游戏场景..."`；  
      - `SceneManager.LoadScene("GameScene");`。  
  - 所有中文日志统一为 UTF-8 简体中文。  

- **目的**：  
  - 让「空槽位创建默认存档」与「已有存档进入游戏」行为完全按设计区分；  
  - 修复闭包和场景名问题，保证多按钮、多槽位场景下行为一致可预期；  
  - 统一 SaveSelectPanel 与整个 UI 面板体系的抽象方式。

## 7. AccountManager（修改）

- **作用**：  
  管理本地账号列表（accounts.json），提供注册/登录/登出与当前账号 Id 获取等能力。  

- **改动前**：  
  - 自行实现单例与 `Awake`，风格与 SaveManager/UIPanelManager 不统一。  
  - Register 方法对参数/内部生成 accountId 的处理不够直观（外部可传 Id，而内部仍用 Guid）。  

- **改动后**（`Assets/Scripts/Services/AccountManager.cs`）：  
  - 修改类声明为 `public class AccountManager : Singleton<AccountManager>`，继承通用单例基类。  
  - `private void Awake()` 仍用于加载账号数据（后续可改为 `protected override void Awake()` 并调用 `base.Awake()`）。  
  - `public bool Register(string _userName, string _password)`：  
    - 校验用户名与密码非空，否则输出 `"[AccountManager] 用户名或密码不能为空"` 并返回 false；  
    - 遍历现有账号，若 userName 已存在，则输出 `"[AccountManager] 该账号或用户名已经存在!"` 并返回 false；  
    - 否则 `accountList.accounts.Add(new AccountEntry(_userName, _password));` 并 Save。  
  - 新增 `public int? GetAccountNum() => accountList?.accounts.Count;`。  

- **目的**：  
  - 与 Singleton 体系接轨，减少重复实现；  
  - 明确注册接口，只允许内部生成 accountId，避免外部传入混乱；  
  - 为 UI 或调试提供账号数量统计能力。

## 8. SaveManager（修改）

- **作用**：  
  管理存档的读写、路径、当前槽位、默认存档创建等。  

- **改动前**：  
  - 同样自带单例实现，与新建的 Singleton\<T\> 不统一。  
  - 无直接统计当前账号已有多少存档的接口。  
  - 点击空槽位时的默认存档创建逻辑散落在 Load/SelectSlot 流程中，UI 不好直接调用。  

- **改动后**（`Assets/Scripts/Services/SaveManager.cs`）：  
  - 修改类声明为 `public class SaveManager : Singleton<SaveManager>`，后续继续沿用原有 Load/Save 行为。  
  - 新增方法 `public int GetCurrentAccountSaveCount()`：  
    - 遍历 `i in [0, SaveSlotCount)`，对每个槽位调用 `HasSaveInSlot(i)`，统计返回 true 的数量；  
    - 可用于 UI 提示「当前账号已有 N 个存档」。  
  - 新增方法 `public void CreateDefaultSaveInSlot(int _slotIndex)`：  
    - 若 `_slotIndex` 越界，则输出警告并返回；  
    - 否则设置 `currentSlotIndex = _slotIndex;` 并调用 `Load()`；  
    - 利用原有 `Load()` 中「若文件不存在则 `CreateDefaultSaveData` + Save」的逻辑，在指定槽位创建默认存档。  

- **目的**：  
  - 将 UI 需要的逻辑（统计已存档数量、在指定槽位创建默认存档）上升到服务层提供统一接口；  
  - 使 SaveSelectPanel 的逻辑更简洁，只关心「有没有存档」「要不要创建默认存档」。

- **作用**：

- **改动前**：

- **改动后**（`Assets/Scripts/Services/AccountEntry.cs`）：  
  - 将注释改为：  
    - `public string accountId; // 账号唯一标识`  
    - `public string password; // 密码`

- **目的**：  
  - 按 code-chinese-format 规则统一中文格式，杜绝乱码；  
  - 提高代码可读性，方便后续维护。
